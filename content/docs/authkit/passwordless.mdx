AuthKit offers two opt-in passwordless paths on the login screen. Both are off by default
and are configured under `passwordless`:

```ts title="config/authkit_server.ts"
passwordless: {
  magicLink: true,    // "Email me a login link"
  passkeyFirst: true, // "Sign in with a passkey" before the password
},
```

Each path also requires the matching capability on the `AccountStore`. With
`lucidAccountStore`, magic link works out of the box (it reuses the password-reset columns);
passkey-first requires the [WebAuthn](/docs/authkit/webauthn) capability
(`withWebauthnCredential`). If a capability is missing, the corresponding option is simply
hidden.

## Magic link

On the password step, AuthKit shows an **"Email me a login link"** button. Submitting it:

1. Issues a short-lived, single-use token (15 minutes). The Lucid store stores it in the
   password-reset columns with an `ml:` prefix — **no migration**. The prefix keeps it
   distinct from a password reset, and consuming a magic link never changes the password.
2. Emails a link to `/auth/interaction/:uid/magic?token=…` (the interaction `uid` is carried
   in the link, so the link resumes the original authorize request). The default mailer is
   used unless you override `mail.onMagicLink`.
3. Always renders **"If the account exists, we sent you a login link"** — regardless of
   whether the account exists, to avoid account enumeration.

Opening the link consumes the token and completes the login with `amr: ['email']`. A
bogus or expired token sends the user back to the start of the login. The request is
throttled by the same login [rate limit](/docs/authkit/account-lockout), and emits
`login.magic_link_sent` / `login.failure` audit events.

### Custom mailer

```ts title="config/authkit_server.ts"
mail: {
  onMagicLink: async ({ email, magicUrl, token }) => {
    // send with your own transport
  },
},
```

## Passkey-first

When `passkeyFirst` is enabled and the looked-up account has at least one passkey, the
password step also offers **"Sign in with a passkey"** — *before* a password is entered.

It reuses the existing passkey ceremony endpoints from the MFA step
(`/auth/interaction/:uid/passkey/options` and `/passkey/verify`), now allowed from the
password stage. A successful assertion completes the login with `amr: ['webauthn']` and
**counts as the strong factor** — no password and no separate MFA step are required.

<Callout type="info">
  Guard rails: the button only appears when the store supports passkeys *and* the account
  has at least one registered credential.
</Callout>
