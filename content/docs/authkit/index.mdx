**AuthKit** turns an AdonisJS app into a full **OpenID Connect / OAuth2 Authorization Server**
(Identity Provider), lets other apps consume it as **OIDC clients**, and gives React frontends
typed auth ergonomics. It is built as an idiomatic wrapper around
[`oidc-provider`](https://github.com/panva/node-oidc-provider) — the battle-tested,
certified OpenID library — so you inherit spec-correct flows while writing only AdonisJS
config and controllers.

<Callout type="info">
  AuthKit handles **authentication** (who you are). Pair it with [Authz](/docs/authz) for
  **authorization** (what you can do) — AuthKit's `Identity` flows straight into Authz's
  `resolveUserRef`, and into the shared [request context](/docs/context).
</Callout>

## What you get

- **OIDC / OAuth2 server** — authorization code + PKCE, refresh tokens with rotation, token
  exchange (RFC 8693), discovery, JWKS, revocation, and introspection — all wired from real
  `oidc-provider` v9 internals.
- **MFA** — [TOTP](/docs/authkit/mfa), [WebAuthn passkeys](/docs/authkit/webauthn), recovery
  codes, and trusted-device skip.
- **Passwordless** — [magic-link email and passkey-first](/docs/authkit/passwordless) login.
- **Protocol extensions** — [Device Flow](/docs/authkit/device-flow) (RFC 8628), DPoP
  (RFC 9449), PAR (RFC 9126), step-up auth via `acr_values`, and Dynamic Client Registration.
- **Multi-tenancy** — [organizations](/docs/authkit/organizations), members, invitations,
  and per-org token claims.
- **Consoles** — an admin console (users, clients, sessions, audit) and a self-service
  account console, both mounted in one call.
- **Tokens & sessions** — [Personal Access Tokens](/docs/authkit/personal-access-tokens),
  admin [impersonation](/docs/authkit/impersonation), and back-channel / RP-initiated logout.
- **Hardening** — progressive [account lockout](/docs/authkit/account-lockout), per-IP
  rate-limiting, an audit log with an events/webhook fan-out, and new-login email alerts.
- **Compliance** — LGPD/GDPR [data export, erasure cascade](/docs/authkit/compliance), and
  security notifications.
- **Operability** — i18n (English + pt-BR), [OpenTelemetry metrics](/docs/authkit/observability),
  `authkit:doctor` / `authkit:rotate-keys` commands, and JWKS keystore vaults for AWS / GCP / Azure.

## Packages

AuthKit ships as a family of `@adonis-agora/authkit-*` packages, so each app pulls in only
what it actually runs:

| Package | Role |
| --- | --- |
| `@adonis-agora/authkit-server` | The OIDC/OAuth2 Authorization Server (Identity Provider). |
| `@adonis-agora/authkit-client` | OIDC relying-party adapter — session auth + JWT/opaque/PAT resolvers. |
| `@adonis-agora/authkit-react` | Frontend ergonomics: a typed `useAuth()` hook + role-gating components. |
| `@adonis-agora/authkit-sdk` | Backend Admin SDK with `remote` (HTTP) and `embedded` (in-process) drivers. |
| `@adonis-agora/authkit-core` | Shared contracts (`Identity`, `SessionResolver`, config types). |
| `@adonis-agora/authkit-testing` | Mint real signed ID tokens in tests. |
| `@adonis-agora/authkit-vault-{aws,gcp,azure}` | JWKS keystore vaults per cloud. |

## Install

Run the Authorization Server in your IdP app:

```bash
node ace add @adonis-agora/authkit-server
```

Consume it from another app as an OIDC client:

```bash
node ace add @adonis-agora/authkit-client
```

Add typed React ergonomics to an Inertia + React frontend:

```bash
npm i @adonis-agora/authkit-react
```

The [Getting Started](/docs/authkit/getting-started) guide walks the complete happy path —
two config files, the host mount, the login/callback/logout controller, and reading the user
in your controllers via `ctx.auth`.

## Two halves: server and client

AuthKit has two cooperating sides. The **server** (`authkit-server`) *is* the Identity
Provider — it owns user identities, mints tokens, and renders the login / consent / account
screens. The **client** (`authkit-client`) makes any AdonisJS app a *relying party*: per
request it resolves an `Identity` from a session JWT, opaque token, or PAT, maps that to your
domain user via `resolveUser`, and exposes `ctx.auth`.

The two can live in the **same** process (embedded) or be **separate** deployments
(standalone IdP + product apps). See [Deployment Topologies](/docs/authkit/topologies) to
choose.

## How it fits the ecosystem

- **[Context](/docs/context)** — the authenticated session populates the ambient request
  context (`userRef`, tenant), so every other Agora library sees the current user.
- **[Diagnostics](/docs/diagnostics)** — AuthKit's audit fan-out publishes auth events
  (`agora:authkit:login.success`, `agora:authkit:mfa.enabled`, `agora:authkit:account.locked`, …)
  on the diagnostics bus.
- **[Telescope](/docs/telescope)** — those events surface automatically in the observability
  console, with an opt-in dedicated **Security** dashboard.
- **[Authz](/docs/authz)** — AuthKit identities feed DB-backed roles & permissions through Bouncer.

See [Agora integration](/docs/authkit/agora-integration) for the full ecosystem wiring.
