Validation in `@agora/filter` happens at two levels: the library's own **structural validation** (always on, runs inside `applyFilter`), and your **application validation** (optional, VineJS â€” for type coercion and business rules). This guide covers both.

---

## Structural validation (built in)

Every time `applyFilter` is given filters, it runs `validateColumnFilters` before applying anything. This guards the shape of each `ColumnFilter`, independent of your allow-list:

- **field** â€” must be a non-empty string matching `^[a-zA-Z_][a-zA-Z0-9_.]*$` (letters, digits, underscores, dots). Anything else is rejected â€” this blocks SQL injection through field names.
- **operator** â€” must be one of the 22 known operators, or a [SQL-symbol alias](/docs/filter/filter/guides/operators#sql-symbol-aliases) (normalized to canonical in place).
- **value shape** â€” matched to the operator:
  - unary operators (`isNull`, `isNotNull`, `isEmpty`, `isNotEmpty`, `exists`, `notExists`) ignore any value;
  - `between`/`notBetween` require a 2-element array;
  - `in`/`notIn`/`isAnyOf` require an array;
  - everything else requires a defined value, and rejects `null` (use `isNull`).
- **depth** â€” `AND`/`OR` nesting beyond `MAX_FILTER_DEPTH` (10) is rejected, preventing stack-overflow DoS from deep payloads.
- **group nodes** â€” a `ColumnFilter` with an empty `field` and an `AND`/`OR` array is treated as a pure boolean group; only its children are validated.

Each failure throws `InvalidColumnFilterError` with a descriptive message.

```ts
import { validateColumnFilters, InvalidColumnFilterError } from '@agora/filter'

try {
  validateColumnFilters([{ field: 'name', operator: 'banana', value: 'x' }])
} catch (err) {
  if (err instanceof InvalidColumnFilterError) {
    // 'Unknown filter operator "banana". Valid operators: equals, notEquals, ...'
  }
}
```

<Callout type="info">
`validateColumnFilter` and `validateColumnFilters` are exported so you can run them manually â€” e.g. before feeding client-derived filters into a `whereHas` sub-query via `applyColumnFilters`, which does **not** validate on its own.
</Callout>

## throwOnInvalid â€” allow-list rejection

Structural validation guards the *shape*; the [allow-list](/docs/filter/filter/guides/filter-classes) guards *which columns*. By default a disallowed field is dropped silently. Set `throwOnInvalid: true` to reject the request instead:

```ts
import { applyFilter, InvalidColumnFilterError } from '@agora/filter'

try {
  applyFilter(query, input, { allowed: ['name'], throwOnInvalid: true })
} catch (err) {
  // InvalidColumnFilterError â€” 'Field "secret" is not filterable.'
}
```

Both structural failures and (when strict) allow-list failures surface as the same `InvalidColumnFilterError`, so you can handle them in one place.

## Turning failures into HTTP 400

In an AdonisJS controller, map `InvalidColumnFilterError` to a `400`:

```ts title="app/controllers/users_controller.ts"
import type { HttpContext } from '@adonisjs/core/http'
import { parseFilterRequest, applyFilter, InvalidColumnFilterError } from '@agora/filter'
import User from '#models/user'

async index({ request, response }: HttpContext) {
  const input = parseFilterRequest(request.qs())
  const query = User.query()

  try {
    const { page, size } = applyFilter(query, input, {
      allowed: ['name', 'email', 'status'],
      throwOnInvalid: true,
    })
    return query.paginate(page, size)
  } catch (err) {
    if (err instanceof InvalidColumnFilterError) {
      return response.badRequest({ message: err.message })
    }
    throw err
  }
}
```

To handle it app-wide, add a branch in `app/exceptions/handler.ts`:

```ts title="app/exceptions/handler.ts"
import { InvalidColumnFilterError } from '@agora/filter'

async handle(error: unknown, ctx: HttpContext) {
  if (error instanceof InvalidColumnFilterError) {
    return ctx.response.badRequest({ message: error.message })
  }
  return super.handle(error, ctx)
}
```

## Layering VineJS

The library validates filter *structure*, not your *domain*. Query-string values arrive as strings, and you may want real numbers/dates, enum constraints, or capped sizes. VineJS â€” AdonisJS's validation layer â€” is the idiomatic place for that. Validate the raw query, then parse:

```ts title="app/validators/list_users.ts"
import vine from '@vinejs/vine'

export const listUsersValidator = vine.compile(
  vine.object({
    filter: vine
      .object({
        status: vine.enum(['active', 'inactive']).optional(),
        age: vine
          .object({
            gte: vine.number().min(0).optional(),
            lte: vine.number().max(150).optional(),
          })
          .optional(),
      })
      .optional(),
    sort: vine.string().optional(),
    search: vine.string().trim().minLength(1).optional(),
    page: vine.number().min(1).optional(),
    size: vine.number().min(1).max(100).optional(),
  })
)
```

```ts title="app/controllers/users_controller.ts"
import { parseFilterRequest, applyFilter } from '@agora/filter'
import { listUsersValidator } from '#validators/list_users'
import User from '#models/user'

async index({ request }: HttpContext) {
  const payload = await request.validateUsing(listUsersValidator)

  const input = parseFilterRequest(payload)          // already coerced + bounded
  const query = User.query()
  const { page, size } = applyFilter(query, input, {
    allowed: ['status', 'age'],
    searchable: ['name', 'email'],
  })
  return query.paginate(page, size)
}
```

VineJS coerces `age.gte` to a real number and rejects out-of-range values with its standard `422` response; `@agora/filter` then handles operator dispatch and allow-listing. The two layers are complementary â€” VineJS owns *value* correctness, the allow-list owns *column* exposure.

<Callout type="info">
VineJS is optional. Without it, `@agora/filter` still parses, structurally validates, and allow-lists â€” values just stay as strings, which SQL comparison tolerates. Reach for VineJS when you need coercion, enums, or per-field bounds.
</Callout>
