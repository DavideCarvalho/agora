The whole server-side surface is two function calls: `parseFilterRequest(request.qs())` to reshape the request, then `applyFilter(query, input, config)` to apply it to a Lucid query and resolve pagination. This guide covers the common controller patterns.

---

## The basic endpoint

```ts title="app/controllers/users_controller.ts"
import type { HttpContext } from '@adonisjs/core/http'
import { parseFilterRequest, applyFilter } from '@agora/filter'
import User from '#models/user'

export default class UsersController {
  async index({ request }: HttpContext) {
    const input = parseFilterRequest(request.qs())
    const query = User.query()

    const { page, size } = applyFilter(query, input, {
      allowed: ['name', 'email', 'age', 'status'],
      searchable: ['name', 'email'],
    })

    return query.paginate(page, size)
  }
}
```

```ts title="start/routes.ts"
import router from '@adonisjs/core/services/router'
const UsersController = () => import('#controllers/users_controller')

router.get('/users', [UsersController, 'index'])
```

## Where input comes from

`parseFilterRequest` takes any decoded object of query params, so you choose the source:

<Tabs items={['Query string (GET)', 'Request body (POST)', 'A nested key']}>
  <Tab value="Query string (GET)">
```ts
// GET /users?filter[status]=active&sort=-createdAt
const input = parseFilterRequest(request.qs())
```
  </Tab>
  <Tab value="Request body (POST)">
```ts
// POST /users/search  with a JSON body in the same Spatie shape
const input = parseFilterRequest(request.body())
```
  </Tab>
  <Tab value="A nested key">
```ts
// Input wrapped under `request.input('query')`
const input = parseFilterRequest(request.input('query', {}))
```
  </Tab>
</Tabs>

<Callout type="info">
A `GET` endpoint pairs with the client builder's `.toQueryString()`; a `POST` search endpoint pairs with `.build()` (the structured `{ filter: { where }, sort, paginate }` body). Both shapes are understood by `parseFilterRequest`. See the [Client Builder](/docs/filter/filter/guides/client).
</Callout>

## A reusable list helper

Most list endpoints are identical apart from the model and the config. Factor the boilerplate into one helper:

```ts title="app/filters/list.ts"
import type { HttpContext } from '@adonisjs/core/http'
import { parseFilterRequest, applyFilter, type FilterConfig } from '@agora/filter'
import type { QueryBuilderLike } from '@agora/filter'

export function list(
  ctx: HttpContext,
  query: QueryBuilderLike & { paginate: (p: number, s: number) => Promise<unknown> },
  config: FilterConfig,
) {
  const input = parseFilterRequest(ctx.request.qs())
  const { page, size } = applyFilter(query, input, config)
  return query.paginate(page, size)
}
```

```ts title="app/controllers/users_controller.ts"
import { list } from '#filters/list'
import User from '#models/user'

export default class UsersController {
  index(ctx: HttpContext) {
    return list(ctx, User.query(), {
      allowed: ['name', 'email', 'status'],
      searchable: ['name', 'email'],
    })
  }
}
```

## Per-request policies

Because `FilterConfig` is a plain object, you can compute it from the request — e.g. widen the allow-list for admins:

```ts
async index({ request, auth }: HttpContext) {
  const input = parseFilterRequest(request.qs())
  const query = User.query()

  const isAdmin = auth.user?.role === 'admin'
  const { page, size } = applyFilter(query, input, {
    allowed: isAdmin
      ? ['name', 'email', 'status', 'internalNotes']
      : ['name', 'status'],
    searchable: ['name', 'email'],
  })

  return query.paginate(page, size)
}
```

You can also pre-scope the query with constraints the client can't override — apply your own `where` before `applyFilter`, and it composes (AND) with the filtered conditions:

```ts
const query = User.query().where('tenantId', auth.user!.tenantId) // forced, not filterable
applyFilter(query, input, { allowed: ['name', 'status'] })
```

<Callout type="warn">
Forced constraints belong on the query **before** `applyFilter`, and the column should be left **out** of `allowed`. That way the tenant scope is always present and a client can never relax it through a filter.
</Callout>

## Strict mode and error handling

Pass `throwOnInvalid: true` to reject queries that reference disallowed fields instead of silently dropping them, and map the error to a `400`:

```ts
import { InvalidColumnFilterError } from '@agora/filter'

async index({ request, response }: HttpContext) {
  const input = parseFilterRequest(request.qs())
  const query = User.query()
  try {
    const { page, size } = applyFilter(query, input, {
      allowed: ['name', 'status'],
      throwOnInvalid: true,
    })
    return query.paginate(page, size)
  } catch (err) {
    if (err instanceof InvalidColumnFilterError) {
      return response.badRequest({ message: err.message })
    }
    throw err
  }
}
```

See [Validation](/docs/filter/filter/guides/validation) for handling the error globally in `app/exceptions/handler.ts`.

## Response shapes

`query.paginate(page, size)` returns Lucid's paginator, which serializes to `{ meta, data }`. You can return it directly, or reshape:

```ts
const result = await query.paginate(page, size)
return {
  data: result.all(),
  meta: result.getMeta(),
}
```

For a non-paginated list (e.g. a small lookup), ignore the returned pagination and run the query directly — the filters, sort, and search are already applied:

```ts
const { /* page, size unused */ } = applyFilter(query, input, { allowed: ['name'] })
return query.exec()
```
